# PatentForge v0.6 — Application Generation + Migration Roadmap

## Overview

v0.6 delivers two things:

1. **Application Generator Service** (Python) — assembles all upstream artifacts into a USPTO-formatted patent application with Word/PDF/Markdown export
2. **Migration Roadmap** — plan to consolidate all services to Python + SQLite for standalone desktop packaging

## Why Python

The endgame is a double-click installer for non-technical users. Python services bundle into self-contained `.exe` files via PyInstaller. Node.js has no production-quality equivalent. Since claim-drafter and compliance-checker are already Python, every new Python service is trivially packaged. Every new TS service adds to the packaging debt.

## Application Generator Service

### Input (gathered by backend adapter)

| Source | Data | Used For |
|--------|------|----------|
| InventionInput | title, description, problemSolved, howItWorks, etc. | Title, Background, context |
| FeasibilityRun stages 1,5,6 | outputText | Abstract seed, Background, Summary, Detail |
| ClaimDraft | claims[], specLanguage | Claims section, Detailed Description |
| ComplianceCheck | results[], overallPass | Validation gate (must pass/warn) |
| PriorArtSearch | results[] | Cross-References section |

### Output Sections (USPTO format)

1. **Title** — from InventionInput.title (cleaned)
2. **Cross-References** — formatted prior art citations
3. **Background of the Invention** — LLM-generated from Stage 1 + problem context
4. **Summary of the Invention** — LLM-generated from Stage 5 + claims overview
5. **Detailed Description** — LLM-generated from spec language + feasibility findings
6. **Claims** — from ClaimDraft (already formatted, just needs numbering)
7. **Abstract** — LLM-generated, 150-300 words, from Stage 1 + claims
8. **Figure Descriptions** — LLM-generated placeholders

All prose sections get USPTO paragraph numbering: `[0001]`, `[0002]`, etc.

### Service Architecture

```

├── src/

```

### LangGraph Pipeline

```
generate_background → generate_summary → generate_detailed_desc → generate_abstract → generate_figures → format_claims → assemble → END
```

Each node calls Claude with a section-specific prompt + all upstream artifacts as context. Nodes run sequentially (each may reference previous sections for consistency). The `assemble` node applies USPTO formatting and produces the final document.

### API Contract

```
POST /generate
Body: {
  "invention_title": str,
  "invention_narrative": str,
  "feasibility_stage_1": str,
  "feasibility_stage_5": str,
  "feasibility_stage_6": str,
  "claims": [{ "claim_number": int, "claim_type": str, "text": str, ... }],
  "specification_language": str,
  "prior_art_results": [{ "patent_number": str, "title": str, "abstract": str }],
  "compliance_passed": bool,
  "settings": {
    "api_key": str,
    "default_model": str,
    "max_tokens": int
  }
}
Response: SSE stream

Events:
  event: step      data: {"step": "background", "status": "generating"}
  event: step      data: {"step": "background", "status": "complete", "content": "..."}
  event: step      data: {"step": "summary", "status": "generating"}
  ...
  event: complete  data: { full ApplicationResult JSON }
  event: error     data: {"error": "..."}

POST /generate/sync
Body: same as above
Response: ApplicationResult JSON directly

GET /health
Response: {"status": "ok", "service": "application-generator", "promptHashes": {...}}

POST /export/docx
Body: ApplicationResult JSON
Response: application/vnd.openxmlformats-officedocument.wordprocessingml.document

POST /export/pdf
Body: ApplicationResult JSON
Response: application/pdf

POST /export/markdown
Body: ApplicationResult JSON
Response: text/markdown
```

### Backend Integration

New module: `backend/src/application/`
- `application.controller.ts` — routes
- `application.service.ts` — orchestrates calls to Python service, saves to DB
- `application.module.ts` — NestJS module

Endpoints:
- `POST /projects/:id/application` — start generation (fire-and-forget pattern)
- `GET /projects/:id/application` — get latest application
- `GET /projects/:id/application/:version` — get specific version
- `PATCH /projects/:id/application/sections/:name` — inline edit a section
- `GET /projects/:id/application/export/docx` — download Word
- `GET /projects/:id/application/export/pdf` — download PDF
- `GET /projects/:id/application/export/markdown` — download Markdown

### Frontend

New component: `frontend/src/components/ApplicationTab.tsx`
- Section navigation (left panel)
- Live preview with paragraph numbering (right panel)
- Inline editing of all sections
- Export toolbar (Word / PDF / Markdown)
- Polling for generation status (3s interval)

Prerequisites: compliance check must be COMPLETE with overallPass=true or only WARN results.

### Docker

Port 3003, added to docker-compose.yml as `application-generator` service.

## Migration Roadmap (v0.7+)

### Goal

Consolidate to a single-runtime stack that can be packaged as a standalone Windows installer:

```
Current (v0.6):                    Target (v0.8+):
  NestJS backend (Node)     →      FastAPI backend (Python)
  Feasibility service (TS)  →      Feasibility service (Python)
  Claim drafter (Python)    ✓      Claim drafter (Python)
  Compliance checker (Python) ✓    Compliance checker (Python)
  App generator (Python)    ✓      App generator (Python)
  PostgreSQL                →      SQLite
  React frontend (Vite)     →      Static files served by backend
```

### Phase 1: v0.7 — SQLite Mode

- Add SQLite support to the NestJS backend via Prisma's SQLite provider
- Controlled by `DATABASE_PROVIDER=sqlite` env var
- PostgreSQL remains default for Docker/cloud deployments
- SQLite file stored at `~/.patentforge/data.db`
- This unblocks standalone mode without rewriting the backend yet

### Phase 2: v0.8 — Port Backend to FastAPI

- Rewrite NestJS controllers/services as FastAPI routes
- Port Prisma ORM to SQLAlchemy or raw SQLite via aiosqlite
- Merge feasibility service into the backend (it's just HTTP calls to Anthropic)
- Single FastAPI process serves: backend API + feasibility + static frontend files
- Frontend build output served from `/static/` route

### Phase 3: v0.9 — Standalone Installer

- PyInstaller bundles: backend + claim-drafter + compliance-checker + app-generator
- Single `.exe` launcher starts all services on localhost ports
- Embedded SQLite (no database to install)
- Frontend is pre-built static files bundled in
- NSIS or Inno Setup for Windows installer
- Creates Start Menu shortcut, opens browser to localhost:8080
- Uninstaller cleans up (preserves user data at `~/.patentforge/`)

### Why Not Migrate Now

v0.6 application generator is the last major feature before the product is functionally complete. Migrating the backend mid-feature would delay shipping. The TS backend works — it's a packaging problem, not a functionality problem. Ship v0.6, then migrate.
